# ADR-001: Replace CLI Dependencies with Embedded Logic

**Status:** Proposed
**Date:** 2026-04-19
**Deciders:** Flavio Gonzalez

## Context

`gofast` currently invokes two external CLI tools at generation time:

- **Atlas CLI** — `exec.Command("atlas", "schema", "inspect", ...)` to dump MySQL schema to SQL
- **sqlc CLI** — `exec.Command("sqlc", "generate")` to generate Go models from SQL queries

Both must be installed on the user's machine and reachable in `$PATH`. This is the only friction point in an otherwise zero-config tool. The goal is to ship `gofast` as a single binary with zero external dependencies.

## Options Considered

### Option A: Keep CLI invocation (status quo)

| Dimension | Assessment |
|-----------|------------|
| Complexity | Low |
| Install friction | High — user needs atlas + sqlc in PATH |
| Maintenance | Low |
| Flexibility | Limited — tied to CLI versions |

**Pros:** Already works. Atlas and sqlc handle edge cases in schema parsing and code generation.

**Cons:** Breaks on any machine without the CLIs installed. Windows PATH issues (already patched with `cmd /C` workaround). Cannot distribute as a single binary.

---

### Option B: Use Atlas Go SDK + sqlc Go SDK

| Dimension | Assessment |
|-----------|------------|
| Complexity | High |
| Install friction | None |
| Maintenance | High — sqlc's programmatic API is internal and unstable |
| Flexibility | Good for Atlas; risky for sqlc |

Atlas SDK (`ariga.io/atlas/sql/mysql`) has a stable, documented Go API for schema inspection.

sqlc's programmatic API (`github.com/sqlc-dev/sqlc/internal/...`) is internal, undocumented, and subject to breakage between minor versions.

**Pros:** No external tool installs. Atlas SDK is well-maintained.

**Cons:** sqlc SDK is not a public API. Coupling to sqlc internals is a maintenance liability.

---

### Option C: Atlas SDK + generate Go models directly (recommended)

| Dimension | Assessment |
|-----------|------------|
| Complexity | Medium |
| Install friction | None |
| Maintenance | Low — only Atlas SDK to track |
| Flexibility | Full control over model generation |

**Key insight:** `gofast` already does the hard part. `schema.SQLLoader` parses tables into typed `schema.Column` structs (`TypeInt`, `TypeString`, `TypeTime`, etc.). This is exactly the input needed to generate Go model files with a template — sqlc is not required.

What sqlc adds for gofast's use case is entirely templatable:
- Struct generation from column definitions → `models.go.tmpl`
- `DBTX` interface and `Queries` struct boilerplate → `db.go.tmpl`
- Typed CRUD query functions → `query_funcs.go.tmpl`

Nullable type mapping is already specified in `sqlc.yaml.tmpl` and can be replicated inline in the new templates.

**Pros:** Single binary, zero install friction, full control, no unstable internal APIs.

**Cons:** Need to implement the three templates above (~200 lines total). One-time cost.

---

### Option D: Embed CLI binaries

| Dimension | Assessment |
|-----------|------------|
| Complexity | Very High |
| Binary size | +20–50 MB |
| Maintenance | High — must vendor and update binaries per platform |

Embed pre-compiled atlas and sqlc binaries for linux/darwin/windows. Extract and execute at runtime.

**Pros:** Guaranteed version compatibility. Truly zero install friction.

**Cons:** Bloated binary, complex build pipeline, licensing concerns. Not suitable for a $0 solo project.

## Decision

**Option C** — Replace Atlas CLI with `ariga.io/atlas/sql/mysql` Go SDK, and replace the sqlc CLI steps with direct template-based model generation.

The sqlc step can be eliminated entirely because `gofast` already parses the schema into typed column data. The output models just need to be templated, not re-derived.

## Consequences

- `gofast` becomes a true single binary — no install instructions needed beyond `go install`
- Atlas SDK adds one dependency; sqlc is removed as a dependency entirely
- Full control over generated model code (nullable handling, naming conventions)
- Pipeline shrinks from 8 steps to 7 (remove `SQLCConfigStep` and `SQLCGenerateStep`, add `ModelsGenerationStep`)
